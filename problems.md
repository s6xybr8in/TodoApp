
## 모델 개선 제안 사항

`lib/models/` 디렉토리의 데이터 모델을 분석한 결과, 다음과 같은 문제점과 개선점을 발견했습니다. 애플리케이션의 안정성과 확장성을 위해 아래 변경 사항들을 적용하는 것을 권장합니다.

### 1. 고유 ID (Unique ID) 생성 방식 변경
- [x] **문제점:**
    - `Daily` 모델: `isSameDay`를 사용하여 날짜를 비교하는데, 이는 Box에 추가할 때 고유성을 보장하는 로직이 모델 외부에 있어 불안정합니다. ID 필드가 별도로 없습니다.
- [x] **개선 제안:**
    - `Daily` 모델: `YYYY-MM-DD` 형식의 날짜 문자열을 고유 ID로 사용하는 것이 좋습니다. 이렇게 하면 특정 날짜의 `Daily` 객체에 직접적이고 안전하게 접근할 수 있습니다. (`Hive.box<Daily>('dailies').get('2026-01-04')`)
    - Daily의 date를 없애고 id로 유일하게 접근할 수 있게 만들기 단. id랑 DataTime 객체랑도 isSameDay를 사용할 수 있어야됨.

### 2. 모델과 데이터 로직 분리 (Repository Pattern)
- [x] **문제점:**
    - `Todo`와 `Daily` 모델 내부에 `Hive.box`에 직접 접근하여 데이터를 수정하거나 추가하는 로직(`markAsDone`, `toggleStar` 등)이 포함되어 있습니다. 이는 모델이 너무 많은 책임(fat model)을 가지게 만들어 테스트와 유지보수를 어렵게 합니다.
- [x] **개선 제안:**
    - 데이터베이스 관련 로직을 별도의 `Repository` 클래스(예: `TodoRepository`, `DailyRepository`)로 분리하는 것을 권장합니다. 모델은 순수하게 데이터 구조만 표현하고, 모든 데이터 CRUD(생성, 읽기, 수정, 삭제) 작업은 `Repository`를 통해 이루어지도록 구조를 변경합니다.

### 3. 불분명하거나 사용되지 않는 속성 정리
- **문제점:**
    - `Todo` 모델의 `className`: '반복 그룹 식별자'라는 주석이 있지만, 이름이 너무 일반적이라 용도를 파악하기 어렵습니다.
    - `Daily` 모델의 `achievement`: `달성률`을 위한 필드로 보이나, 현재 앱 내에서 이 값을 계산하거나 사용하는 로직이 없습니다.
- **개선 제안:**
    - `className`: `repetitionGroupId` 와 같이 용도를 명확하게 나타내는 이름으로 변경합니다.
    - `achievement`: 이 필드를 어떻게 계산하고 표시할지 구체적인 계획을 세우고 구현하거나, 사용하지 않을 경우 모델에서 삭제하여 코드를 명확하게 유지합니다.
